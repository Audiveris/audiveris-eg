Audiveris

Updated:

Current version is 3.0, with a beta release published on 5-Jul-2007. With this version, Audiveris can now be considered as a fully-fledged music scanner, going all the way from the scanned music sheet to the MusicXML output.

Project Goals

Audiveris is an Optical Music Recognition (OMR) module. It is a music scanner: starting from the simple bitmap of a music sheet (for example by scanning a paper), it is able to infer the music information the sheet contains. It can then provide high-level logical music information to other tools such as a Midi Sequencer, or a Composition Editor, or any other tool that complies with the MusicXML definition.

Audiveris works with printed music sheets only, the task of recognizing hand-written scores is at least one order of magnitude harder.

There are already commercial tools in this area, such as MusicScan or SharpEye. Audiveris is, to our knowledge, the first Java open-source OMR tool. it is also a cross-platform tool, since developed entirely in Java, and tested on Windows, Solaris, Mac OS (on-going) and Linux (soon).

There is still room for improvement in this program (see below the foreseen contributions), but the internal architecture is now fairly stable and robust enough to be used and augmented by people willing to contribute.

Current Status

Starting with version 3.0, Audiveris is now a complete music scanner which aims at automatically translating the 'pixel data' of a sheet of music to the high-level music information carried by a MusicXML file. Since the translation cannot be perfect, since there may still be some errors in the translation, the program offers the user the ability to guide the translation on the fly. It can also use new translation data in order to further train its internal neural network, in order to improve the glyph recognition task.

More detailed information is available in the provided manuals, but here are the main phases of the application:

  1. Detection and correction of potential skew angle of the sheet,
  2. Removal of staff lines, detection of systems, staves and measures,
  3. Extraction of individual music items, essentially based on their appearance and their structure,
  4. Music translation based on the extracted music items, and essentially driven by music grammar,
  5. Manual correction of unavoidable remaining errors,
  6. Export of logical music information, using the MusicXML format definition,
  7. Finally, any other application able to read / write MusicXML format can be chained to it.

Audiveris is the result of several years of development at night, and was initially based on the lessons learnt from a prototype written in Ada several years ago. Development is still on going, and would benefit from additional contribution. We would also appreciate feedback of other users, in the form of suggestions or bug reports (please provide the scanned image with your report, to help us analyze the program behavior).

Foreseen Contributions

Here is a list of domains and tasks for which contribution would make sense and be really appreciated :

  • Rather easy
    • Validate Audiveris on other platforms than Windows or Solaris. Some people volunteered for Mac OS (Brenton Partridge is actively working on it) and Linux.
  • More significant
    • Integrate with a Visual Music Editor for manual correction, for printing, for playing, etc... For example Xemo (project as seen on NetBeans) would be a natural partnering tool, however the Xemo site doesn't seem active anymore. Audiveris has only an embedded rudimentary editor. So this task remains fully open.
    • Find and integrate an open-source OCR (Optical Character Recognition), preferably in Java, in order to handle the various text pieces (typically lyrics) which for the time being are simply discarded.
    • Evaluate a move to NetBeans (The platform, not the IDE). This would move Audiveris stand-alone application to a much better integration environment, taking advantage of view and parameter management. This should also ease the integration with visual editors.
  • Fairly difficult
    • Consider addressing hand-written scores . This domain could take benefit of a significant part of Audiveris framework, however this would really be a sub-project on its own. To be discussed if anyone is interested by this daunting task.

Current Implementation

Audiveris today is a pure Java stand-alone application, which represents about 260 classes and 85_000 lines of Java code.

Techniques
  • Linear Adjacency Graphs (LAG), a technique to record pixel information as a directed graph structure, composed of pixel contiguous runs organized in sections linked by junctions.
  • Sticks, a technique developed above LAGs to specifically take advantage of the structuring sticks of a music sheet (staff lines, ledgers and legato signs are examples of horizontal sticks, while stems and bar lines are examples of vertical sticks).
  • Neural Network, to allow efficient pattern recognition as well as training based on user input for continuous improvement of the tool efficiency.
Technologies
  • Java Advanced Imaging (JAI) for number crunching in image processing (loading image, accessing pixel data, rotating the image, etc)
  • Swing for user interface, with JGoodiesenhancements in terms of forms and look & feel.
  • JFreeChart for charts.
  • JAXB 2 for XML persistency.
  • JAMA for linear algebra in Java.
  • Java 5 since generics are used wherever the compiler can help, and preferably Java 6 since JAXB 2 is already integrated with Java 6 distribution.
  • Ant for project building.
Companions
  • ProxyMusic is a companion project, dedicated to the marshalling and unmarshalling between Java objects and MusicXML files. Audiveris today uses ProxyMusic V1.1 (since compliant with MusicXML V1.1) and will soon support MusicXML V2.0 which Recordare has just released. Please refer to https://proxymusic.dev.java.net
